# 运行状态告警处理闭环

- 用于监控状态异常
- 某一状态从正常到异常再恢复正常是一个闭环

## 核心概念

### 工况事件

- 原始数据点，高频且无状态。

### 状态表

- 在原始工况间传递必要的持久化数据，以便完成告警闭环。
- 维护持续性条件（如指标连续超阈值时长）

### 异常

- 对于关注的每一项指标，基于状态表（或者状态表和工况事件），计算，得到异常表
- 异常表中的每个异常包含一个联合主键。

### 告警

- 一个异常（以联合主键为准）对应一个告警。
- 告警有 `ACTIVE`、`RESOLVED`、`SILENCED` 三个主状态。

## 数据流

1. 收到工况事件
2. 更新状态表
3. 计算异常表
4. 计算告警变化
   - 扫描所有 `ACTIVE`，对照异常表，如果异常主键消失，转化为 `RESOLVED`
   - 扫描所有异常主键，对照告警表，如果没有该主键或者主键为 `RESOLVED`，创建 `ACTIVE` 或者变为 `ACTIVE`；如果有主键且主键为 `SILENCED`，不管。
5. 告警信息落盘（包括状态转换等）
6. 向所有前端推送告警变化
   - 只推送新增 `ACTIVE`，`RESOLVED->ACTIVE`，`ACTIVE->RESOLVED`

## 告警静默

- 前端提供静默和取消静默，静默为无条件变为 `SILENCED`，取消静默为无条件变为 `RESOLVED`。
- 静默后异常消失，不需要转化为 `RESOLVED`，保持 `SILENCED` 即可。

## 潜在缺陷

- 收不到工况事件（告警无变化可以接受）
- 历史记录膨胀（可接受）
- 系统重启导致状态表丢失（期望情况下，重启后的告警状态直接基于首次工况汇报后的初始状态表计算即可，ACTIVE/RESOLVED/SILENCED 都是接受的）

## 风险点排除

- 事件乱序（不考虑）
- 状态表、异常表计算性能问题（可接受）
- 告警抖动、恢复判定问题（状态表用于处理抖动）
