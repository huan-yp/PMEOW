# 任务领域：状态收敛与事件生成模型

## 归属

- 任务的状态收敛、diff、事件生成、落库编排、查询和控制统一归 `packages/core/src/task/` 领域层。
- `TaskEngine`（`task/engine.ts`）是任务摄入的唯一业务入口，负责从一份 Agent 汇报中产出任务状态变化并执行落库。
- `TaskService`（`task/service.ts`）是任务查询和控制的唯一对外入口。
- `diffTasks`（`task/differ.ts`）是任务 diff 算法，基于上一轮和本轮的 active/recentlyEnded 集合计算状态变化。
- `TaskEvent` / `TaskEventType`（`task/events.ts`）定义任务事件类型。
- `IngestPipeline`（`ingest/pipeline.ts`）仅负责编排：把 `UnifiedReport` 分发给 `TaskEngine`、告警引擎和安全分析，不再直接持有任务领域规则。

## 核心原则

- Agent 定时上报里的 `queued`、`running`、`recentlyEnded` 是任务状态的唯一事实来源。
- 任务信息更新和任务事件生成是同一次 diff 的两个输出。
  - 根据 diff 结果逐条处理：active 任务走 upsert，ended 任务走 endTask。
  - 每条 diff 处理完毕后立即生成对应事件。
- `taskId` 是唯一键；如果一个任务已经进入终态，就不应该再以同一个 `taskId` 回到 active 态。

## 状态机约束

- active 态只允许两种：`queued`、`running`
- terminal 态只允许一组结束态：`succeeded`、`failed`、`cancelled`、`abnormal`
- 合法迁移：
  - `queued -> queued`
  - `queued -> running`
  - `queued -> terminal`
  - `running -> running`
  - `running -> terminal`
- 不允许的迁移：
  - `terminal -> queued/running`
  - `running -> queued`

## 任务信息更新机制

- 直接覆盖写的字段
  - 当前状态：`status`
  - 调度信息：`priority`、`scheduleHistory`
    - `scheduleHistory` 直接整段覆盖，不做 patch 合并。
  - 运行态信息：`startedAt`、`pid`、`assignedGpus`、`declaredVramPerGpu`
- 首次写入后通常保持稳定的字段
  - `command`、`cwd`、`user`、`launchMode`
  - `requireVramMb`、`requireGpuCount`、`gpuIds`
  - `createdAt`
- 只在终态落库时写入的字段
  - `finishedAt`、`exitCode`、`endReason`
  - 如果 Agent 显式给出 `recentlyEnded`，优先采用 Agent 给出的结束信息。
  - 如果任务只是从 active 集合里消失，就合成一个 `abnormal/disappeared` 的结束记录。

## 基于状态上报的 diff 过程

- 输入
  - 上一轮 active 集合：上一轮汇报中的 `queued + running`
  - 当前 active 集合：这一轮汇报中的 `queued + running`
  - 当前显式结束集合：这一轮汇报中的 `recentlyEnded`
- 处理中统一按 `taskId` 建索引。
- 根据 diff 结果逐条分支处理：
  - active 任务 → `upsertTask` 覆盖写当前信息
  - ended 任务 → `endTask` 写入终态字段
- 对结束任务分两类判定：
  - 显式结束：出现在 `recentlyEnded`
  - 隐式结束：上一轮在 active，这一轮既不在 active，也不在 `recentlyEnded`
- 对结束任务直接更新终态字段，不需要先构造中间状态。

## 任务事件生成机制

- 事件共三类：`submitted`、`started`、`ended`
- `submitted`
  - 定义：任务第一次出现在 active 集合中。
  - 包括首次出现在 `queued` 或首次出现就已经是 `running`。
  - 如果首次出现就是 `running`，会同时生成 `submitted` 和 `started`。
- `started`
  - 定义：任务进入 `running` 状态。
  - 包括两种情况：
    - 上一轮在 `queued`，这一轮变成 `running`
    - 上一轮完全没出现，这一轮第一次出现就是 `running`（此时伴随 `submitted`）
- `ended`
  - 定义：任务在这一轮被判定进入终态。
  - 包括两种情况：
    - 出现在 `recentlyEnded`（显式结束）
    - 上一轮在 active，这一轮从汇报里消失（隐式消失，合成 `abnormal/disappeared`）

## 数据流

```text
收到一轮任务状态上报
→ IngestPipeline 取出上一份 report
→ 调用 TaskEngine.processReport(serverId, prevReport, currentReport)
→ TaskEngine 内部：
  → 提取上一轮和本轮的 active 视图
  → 构造 recentlyEnded 视图
  → 调用 diffTasks 生成 TaskDiffResult[]
  → 逐条处理：active 走 upsertTask，ended 走 endTask
  → 返回 TaskEvent[]
→ IngestPipeline 遍历事件并通过 onTaskEvent 回调广播
→ 对外广播事件
```
