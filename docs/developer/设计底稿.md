# 设计

## Agent 职责

### 管理和调度本地任务

- 调度本地任务（利用基于队列的历史窗口中的资源信息）
- 维护资源占用信息
  - 部分进程的信息应该过滤
  - GPU 的资源占用需要基于任务队列信息计算，而非单纯探针状态。
- 维护任务队列信息（排队中，运行中，已结束），每个任务的需求信息，被调度信息都需要这里维护。
  - 被调度信息就是计算当前资源是否满足其需求，其是否应该被调度，只维护最近 5 次。

### 汇报信息

汇报硬件资源状态信息（对于 GPU 含任务预留情况等）。

汇报任务队列和任务信息。（已结束的可以扔掉，只汇报排队中和运行中的）

两条线走统一定时汇报。

### 对外接口

- 取消任务。
- 调整任务优先级。

## Agent 架构

- `collector/`
  - 给定 `task_queue` 实例，生成硬件资源状态信息。
  - 入口点 `collect_snapshot`
- `queue/`
  - `scheduler.py`：调度逻辑部分
  - `history.py`：维护 GPU 历史窗口信息的数据结构。
- `state/`
  - 主要维护 `task_queue` 实例
- `daemon/`
  - `service.py`：总控编排
  - `runtime_monitor`：进程生命周期管理。
  - 
- `executor/`
  - 实际执行任务相关
- `transport/`
  - 和 Web 模块通信

## Agent 数据/控制流

### Task 控制流

```
submit
→ queued
→ collect_cycle 评估
→ scheduler 结合当前快照和历史窗口做准入
→ reserve GPU
→ 启动或等待 attached 确认
→ running
→ runtime_monitor/executor 监控任务汇报状态变动
→ task_queue 回收
```

### 状态数据流

```
→ GPU 卡遥测 + GPU 进程
→ GPU 进程和任务队列对齐
→ 得到 per_gpu 分配摘要
→ 反构造最终 gpu_cards 报告
→ 合并 CPU/内存/磁盘/网络/进程/本地用户
→ 得到 resource_snapshot
→ 同 tick 内把任务队列转成 task_snapshot
→ reporter 封装成 UnifiedReport
→ transport 上报
```

## Agent 难点

#### 任务状态定义

- 占用资源
  - 排队中（非法）不可能出现
  - 运行中（合法）
  - 已结束（非法）不可能出现
- 不占用资源：
  - 排队中（合法）
  - 运行中（非法）发生异常，转入已结束（异常）
  - 已结束（合法）
    - 被取消
    - 已完成
    - 异常

#### 基于状态上报的任务状态收敛模型

核心原则

- Agent 定时上报里的 `queued`、`running`、`recentlyEnded` 是任务状态收敛的唯一事实来源。
- 任务信息更新和任务事件生成是同一次 diff 的两个输出。
  - 先收敛任务表里的状态。
  - 再基于同一份 diff 结果生成事件。
- 事件不是写库的输入，不能走“先产事件，再靠事件反推任务状态”的链路。
- `taskId` 是唯一键；如果一个任务已经进入终态，就不应该再以同一个 `taskId` 回到 active 态。

状态机约束

- active 态只允许两种：`queued`、`running`
- terminal 态只允许一组结束态：`succeeded`、`failed`、`cancelled`、`abnormal`
- 合法迁移：
  - `queued -> queued`
  - `queued -> running`
  - `queued -> terminal`
  - `running -> running`
  - `running -> terminal`
- 不允许的迁移：
  - `terminal -> queued/running`
  - `running -> queued`

任务信息更新机制

- 直接覆盖写的字段
  - 当前状态：`status`
  - 调度信息：`priority`、`scheduleHistory`
    - `scheduleHistory` 直接整段覆盖，不做 patch 合并。
  - 运行态信息：`startedAt`、`pid`、`assignedGpus`、`declaredVramPerGpu`
- 首次写入后通常保持稳定的字段
  - `command`、`cwd`、`user`、`launchMode`
  - `requireVramMb`、`requireGpuCount`、`gpuIds`
  - `createdAt`
  - 这些字段在 active 阶段仍可继续幂等 upsert，但不需要为变化单独生成事件。
- 只在终态落库时写入的字段
  - `finishedAt`、`exitCode`、`endReason`
  - 如果 Agent 显式给出 `recentlyEnded`，优先采用 Agent 给出的结束信息。
  - 如果任务只是从 active 集合里消失，就合成一个 `abnormal/disappeared` 的结束记录。

基于状态上报的 diff 过程

- 输入
  - 上一轮 active 集合：上一轮汇报中的 `queued + running`
  - 当前 active 集合：这一轮汇报中的 `queued + running`
  - 当前显式结束集合：这一轮汇报中的 `recentlyEnded`
- 处理中统一按 `taskId` 建索引。
- 对当前 active 集合里的每个任务直接做 upsert，用于收敛任务当前信息。
- 对结束任务分两类处理：
  - 显式结束：出现在 `recentlyEnded`
  - 隐式结束：上一轮在 active，这一轮既不在 active，也不在 `recentlyEnded`
- 对结束任务直接更新终态字段，不需要先构造中间状态。

任务事件生成机制

- 事件只保留三类主事件：`submit`、`start`、`end`
- `submit`
  - 定义：这一轮相对上一轮新增进入 `queued` 的任务。
  - 含义：任务第一次被观测到处于排队态。
- `start`
  - 定义：这一轮相对上一轮新增进入 `running` 的任务。
  - 包括两种情况：
    - 上一轮在 `queued`，这一轮进了 `running`
    - 上一轮完全没出现，这一轮第一次出现就是 `running`
  - 这意味着系统不强求 `submit` 一定先于 `start`。
- `end`
  - 定义：任务在这一轮被判定进入终态。
  - 包括两种情况：
    - 新增进入 `recentlyEnded`
    - 上一轮在 `queued/running`，这一轮从汇报里消失

不单独出事件、只做状态收敛的变化

- `scheduleHistory` 更新
- `priority` 更新
- `pid` 更新
- `assignedGpus` 更新
- `declaredVramPerGpu` 更新
- 这类变化对任务详情很重要，但不一定值得作为单独事件扇出；直接写库即可。

建议处理顺序

```text
收到一轮任务状态上报
→ 取上一轮 active 视图
→ 构造本轮 queued/running/recentlyEnded 视图
→ 先按本轮 active 任务 upsert 当前信息
→ 再处理显式结束和隐式消失，写入终态字段
→ 基于同一份 diff 结果生成 submit/start/end 事件
→ 对外广播事件
```

这样分层之后：

- 任务表承担“最新任务事实”的职责。
- 任务事件承担“需要通知外部发生了什么”的职责。
- 二者共享同一个 diff 过程，但不要互相驱动。

### 事件回调

- 收到两类 Agent 汇报的时候调用有关的回调

- 收到资源信息

  - 向有关 Websocket Web/APP 推送资源信息更新
  - 根据自定义的资源空闲等钩子给有关的前端（Websocket Web/APP）推送事件。

- 收到当前任务队列信息

  - diff 一下，找到哪些已经开始运行了，哪些已经结束了（上一轮汇报过，这一轮没了）
  - 已经结束的不需要管为什么结束的，直接标记为已结束。

  - 给有关的前端（Websocket Web/APP，QQ 等）推送有关的事件。
