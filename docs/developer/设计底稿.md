# 设计

## Agent 职责

### 管理和调度本地任务

- 调度本地任务（利用基于队列的历史窗口中的资源信息）
- 维护资源占用信息
  - 部分进程的信息应该过滤
  - GPU 的资源占用需要基于任务队列信息计算，而非单纯探针状态。
- 维护任务队列信息（排队中，运行中，已结束），每个任务的需求信息，被调度信息都需要这里维护。
  - 被调度信息就是计算当前资源是否满足其需求，其是否应该被调度，只维护最近 5 次。

### 汇报信息

汇报硬件资源状态信息（对于 GPU 含任务预留情况等）。

汇报任务队列和任务信息。（已结束的可以扔掉，只汇报排队中和运行中的）

两条线走统一定时汇报。

### 对外接口

- 取消任务。
- 调整任务优先级。

## Agent 架构

- `collector/`
  - 给定 `task_queue` 实例，生成硬件资源状态信息。
  - 入口点 `collect_snapshot`
- `queue/`
  - `scheduler.py`：调度逻辑部分
  - `history.py`：维护 GPU 历史窗口信息的数据结构。
- `state/`
  - 主要维护 `task_queue` 实例
- `daemon/`
  - `service.py`：总控编排
  - `runtime_monitor`：进程生命周期管理。
  - 
- `executor/`
  - 实际执行任务相关
- `transport/`
  - 和 Web 模块通信

## Agent 数据/控制流

### Task 控制流

```
submit
→ queued
→ collect_cycle 评估
→ scheduler 结合当前快照和历史窗口做准入
→ reserve GPU
→ 启动或等待 attached 确认
→ running
→ runner 或 runtime monitor 回收
→ 从活跃队列移除
```

### 状态数据流

```
→ GPU 卡遥测 + GPU 进程
→ GPU 进程和任务队列对齐
→ 得到 per_gpu 分配摘要
→ 反构造最终 gpu_cards 报告
→ 合并 CPU/内存/磁盘/网络/进程/本地用户
→ 得到 resource_snapshot
→ 同 tick 内把任务队列转成 task_snapshot
→ reporter 封装成 UnifiedReport
→ transport 上报
```



## Agent 难点

#### 任务状态定义

- 占用资源
  - 排队中（非法）不可能出现
  - 运行中（合法）
  - 已结束（非法）不可能出现
- 不占用资源：
  - 排队中（合法）
  - 运行中（非法）发生异常，转入已结束（异常）
  - 已结束（合法）
    - 被取消
    - 已完成
    - 异常

### 持久化存储

#### 状态追踪

窗口策略

- 短期窗口为 1min，保留最近 30 份状态快照

- 永久窗口粒度为 30min，预期一天保留约 50 份状态快照。

归档策略

- 同一个窗口里直接用最新那一份就行。

#### 告警管理

- 同一告警就维护一个最新的（同机器，同类型）

- 忽略管理正常维护

#### 任务队列

根据节点汇报的任务队列信息合理维护，用任务 id 作为唯一编号。

任务运行中（running）的资源占用需要去状态记录窗口中用 pid 查询。

### 事件回调

- 收到两类 Agent 汇报的时候调用有关的回调

- 收到资源信息

  - 向有关 Websocket Web/APP 推送资源信息更新
  - 根据自定义的资源空闲等钩子给有关的前端（Websocket Web/APP）推送事件。

- 收到当前任务队列信息

  - diff 一下，找到哪些已经开始运行了，哪些已经结束了（上一轮汇报过，这一轮没了）
  - 已经结束的不需要管为什么结束的，直接标记为已结束。

  - 给有关的前端（Websocket Web/APP，QQ 等）推送有关的事件。

### 管理人员

- 记录一下人员的资源占用情况（直接基于机器用户聚合。
- 记录一下人员的任务情况。

### 节点状态和历史

#### 显示 UI

- 自适应折线图模式
  - 每个 GPU 的显存占用（按人员归属）
  - 其它信息变化趋势图（注意
  - 所有折线图基本能复用同一套基础模式，前端只请求一次历史，做缓存，然后根据拿到的最新推送只更新最新一个窗口的情况
    - 提供一个按钮刷新历史缓存（主要做调试用）
  - 关于数据点跨度要能够自适应之类的。
- 横条模式（用于每个 GPU 当前的占用/预留情况）
  - 按人员分组涂色

#### 概览

- 实时折线图功能

- 历史回放
  - 概览默认为当前时间点，可以改，改了之后几个折线图都显示历史情况（历史那个时刻的概览情况，详细度可以降低）

#### 历史

- 历史查询
- 节点详情的任务版面只写个大概，主要 tab 放到任务

#### 进程

- 默认展示当前的情况，提供基础的排序功能
- 可以选择一个时间点看那个时候的快照（如果保留了窗口）
- 可以选择一个 PID 去追溯这个进程的生命周期和占用（可以考虑复用折线图）
