# 常见问题与排障

这份文档按症状整理常见问题。优先级从“服务起不来”到“数据不对”再到“任务行为不符合预期”。

## Web 服务端起不来

### 症状

- `pnpm start:web` 直接退出
- 浏览器无法打开 `17200`
- Docker 容器启动后立即重启

### 先检查什么

1. 是否已经先执行过 `pnpm build:web`
2. `PORT` 是否被其他进程占用
3. `MONITOR_DB_PATH` 指向的目录是否可写
4. 如果是 Docker，容器内 `/data` 是否可写

### 说明

当前生产模式入口是 `packages/web/dist/server.js`。如果没有先构建，`start:web` 不会自动替你编译。

## 登录后很快又回到登录页

### 常见原因

- 服务端重启了
- 没有配置稳定的 `JWT_SECRET`

### 处理方式

在正式部署中固定 `JWT_SECRET`，避免每次重启都生成新的临时签名密钥。否则旧 token 会立即失效，前端会表现成重新登录。

## SSH 节点测试连接失败

### 常见原因

- `host`、`username`、端口填写错误
- 服务端进程无法读取指定私钥
- Docker 场景下忘了挂载 `~/.ssh`

### 处理方式

1. 先在“服务器管理”页点击“测试连接”。
2. 确认目标主机可以从服务端所在机器访问。
3. 确认密钥路径真实存在且权限正确。
4. 如果使用上传密钥，确认文件已经进入工作目录下的 `data/keys/`。

## Agent 节点已启动，但 Web 上看不到绑定

### 先检查这几个条件

1. `PMEOW_SERVER_URL` 是否指向正确的 Web 基础地址
2. 节点 hostname 是否和 Web 端服务器记录中的 `host` 精确一致
3. 是否存在多条使用同一 hostname 的服务器记录
4. daemon 是否持续运行，心跳是否仍在发送

### 关键事实

- Agent 绑定不是模糊匹配，而是 hostname 精确匹配。
- 如果重复服务器记录导致匹配不唯一，自动绑定不会成功。
- Agent 心跳是维持 live session 的必要条件，session 超时后服务端会认为 Agent 离线。

## Tasks 页面有数据，但任务一直停在 queued

### 常见原因

- 队列当前被暂停
- 没有满足要求的 GPU 数量或可用显存
- 资源只在瞬时满足，但历史窗口内不持续满足
- 目标节点上的 daemon 已退出或无法启动子进程

### 处理方式

1. 先看 Tasks 页面对应节点是否处于暂停状态。
2. 再看服务器详情里的 GPU allocation 和进程审计。
3. 在节点本地执行 `pmeow-agent status` 和 `pmeow-agent logs <task_id>`。
4. 核对提交任务时的 `--pvram` 和 `--gpu` 是否过于激进。

要特别注意，Agent 调度不仅看当前瞬时资源，还会看历史窗口数据。默认窗口为 120 秒。

## 看不到 GPU 指标或 GPU 一直显示 N/A

### 常见原因

- 节点上没有 NVIDIA GPU
- `nvidia-smi` 不可用
- 驱动异常或容器里没有正确透传 GPU 设备

### 处理方式

先在节点本地确认 `nvidia-smi` 是否可执行，再确认 Agent 进程运行用户是否能读取相关信息。

## Security 里把事件标记为安全后，问题还在

这是当前设计的正常行为。`mark safe` 只会把安全事件记录标记为已处理，不会：

- 杀掉进程
- 解除 GPU 占用
- 自动修改阈值

如果事件本身仍然持续发生，后续仍可能生成新的记录。

## 我该去哪里看真实数据和日志

### 服务端

- SQLite 数据库：`MONITOR_DB_PATH` 指向的位置，或默认工作目录下的 `data/monitor.db`
- 上传密钥目录：工作目录下 `data/keys/`
- 运行日志：当前主要是前台进程输出或容器日志

### Agent 节点

- 本地状态目录：默认 `~/.pmeow/`
- 本地数据库：`~/.pmeow/pmeow.db`
- 控制 socket：`~/.pmeow/pmeow.sock`
- 任务日志目录：`~/.pmeow/logs/`

## 什么时候适合直接重启

下面这几类问题，直接重启通常比长时间猜测更快：

- 修改了 Web 环境变量后希望它们生效
- Agent daemon 已经明显失去连接状态
- Docker 容器或本地进程在升级后加载了旧配置

但如果你怀疑是 hostname 冲突、数据库污染或队列资源判断错误，应该先看数据再重启。重启只能清 session，不能修复错误配置。

## 最后一个排障顺序建议

如果你不知道该从哪里开始，按下面顺序通常最省时间：

1. 先确认进程是否真的在运行。
2. 再确认 URL、端口、路径和权限。
3. 然后确认节点是否真的绑定到正确的服务器记录。
4. 最后再怀疑调度、告警或安全规则本身。
