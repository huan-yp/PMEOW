# 人员与访问管理

这份文档聚焦一个可选但已经落地的能力层：人员档案、服务器用户名绑定，以及个人移动端访问令牌。

如果你只是想先把监控和节点接起来，可以先跳过这页。PMEOW 在没有任何人员数据时仍然可以正常监控、调度和记录安全事件。

## 什么时候需要这页

下面这些场景应该优先看这里：

- 你希望把任务和 GPU 使用归属到真实人员，而不是只看到系统用户名。
- 你希望把同一个人跨节点的多个系统用户名聚合起来。
- 你需要给实验室成员发放个人移动端访问令牌。

## 三个核心对象

### 人员档案

人员档案描述的是“这个人是谁”，包括：

- 显示名称
- 邮箱
- QQ
- 备注

### 绑定关系

绑定关系描述的是“这个人在哪台服务器上对应哪个系统用户名”。一个人可以有多条绑定关系，例如：

- `gpu-1 · alice`
- `gpu-2 · alice-lab`

### 个人移动端令牌

个人移动端令牌描述的是“这个人是否可以使用 `/m/me` 这一套个人移动端入口”。它和管理员 JWT 不是同一套认证材料。

## 人员目录

桌面端“人员”页是入口页面。当前目录会显示：

- 当前显存占用
- 运行中任务数
- 排队任务数
- 活跃节点数

这里的目标不是取代任务页，而是给管理员一个“按人而不是按节点看系统”的入口。

## 创建人员向导

当前新增人员走的是向导式流程，入口在 `/people/new`。入口页现在支持三种开始方式：

- 从服务器用户开始
- 手动创建
- 一键添加未归属用户

### 从服务器用户开始

适合你已经在节点上观测到了某个系统用户名，希望把它快速变成一条正式人员记录。

当前流程会经过：

1. 选择一个种子系统用户。
2. 填写人员档案。
3. 选择需要附加的其他绑定候选项。
4. 最后统一确认。

系统会根据相同或相近的用户名给出绑定建议，帮助你把同一个人的多个账号一起纳入。

如果候选账号很多，seed 列表现在支持分页，不需要再滚很长的列表才能继续下一步。

### 手动创建

适合这个人还没有明显工作负载，或者你想先建档，再等待后续补绑定关系。

这种模式下可以不带任何绑定先创建人员档案。

### 一键添加未归属用户

适合节点里已经积累了很多未归属用户名，希望先按“同名即同人”的规则快速做一轮基础建档。

当前规则如下：

1. 只处理当前还没有绑定到任何人员的用户名。
2. 用户名精确等于 `root` 的账号会自动跳过。
3. 如果系统里存在唯一同名人员档案，则复用该人员，并把同名未归属账号全部绑定过去。
4. 如果系统里不存在同名人员档案，则自动创建一条新人员档案，显示名称直接使用用户名。
5. 如果系统里存在多条同名人员档案，则不会自动选择其中一条，而是跳过并记入报告，等待人工处理。

执行完成后，页面会停留在结果报告页，而不是直接跳走。报告页会展示：

- 汇总统计：处理了多少用户名、创建了多少人员、复用了多少人员、创建了多少绑定、跳过了多少 root、跳过了多少同名冲突
- 详情列表：每个用户名最终是“已创建人员”“复用已有人员”“已跳过 root”“同名冲突”“已被绑定”还是“处理失败”
- 可翻页查看的账号明细：每个用户名实际涉及了哪些节点账号

### 为什么有时最后一步不能提交

如果你选中了一个已经绑定到其他人员的系统用户名，向导会要求你显式勾选“我确认转移已绑定账号”。当前实现会先禁用旧绑定，再创建新绑定，避免误操作直接覆盖别人已有的归属关系。

## 人员详情页

点击人员目录中的任意卡片后，会进入人员详情页。当前详情页包含四块信息：

- 基本信息：显示名称、邮箱、QQ、备注
- 绑定关系：哪些服务器用户名当前属于这个人，哪些绑定已禁用
- 显存时间线：该人在最近时间窗口内的 VRAM 使用聚合
- 关联任务：服务端镜像里与这个人关联的任务列表

这页同时也是个人移动端令牌的管理入口。

## 历史统计周期

人员详情页在绑定和时间线卡片上方提供 **24h / 7d / 30d** 三个周期切换标签。切换标签后，以下数据会按所选周期重新加载：

- **VRAM 占用时长**：该人员在所选周期内持有 GPU 显存的累计时长（小时）。每次快照中只要该人员正在占用 VRAM，就计入一个快照间隔的时长。
- **GB·h**：该人员的 VRAM 容量-时间积累。用于区分"少量显存持续占用"和"大量显存短暂占用"两种模式。计算方式为每次快照的 VRAM 占用量（GB）乘以快照间隔再累加。
- **当前显存**：周期内最近一次快照的 VRAM 占用量。
- **运行任务数**：周期内仍在运行的任务数。
- **显存时间线**：按所选周期显示的 VRAM 使用曲线。
- **关联任务**：周期内与该人员关联的任务列表。

默认周期为 7 天。如果所选周期内没有数据，汇总卡片显示占位符而不是零值。

## 个人移动端令牌生命周期

当前管理员在人员详情页可以执行：

- 创建令牌
- 轮换令牌
- 吊销令牌
- 查看令牌是否存在以及最近使用时间

需要记住两条边界：

- 明文令牌只会在创建或轮换的当下显示一次。
- 个人令牌只用于 `/m/me` 和 `/api/mobile/me/*`，不等于管理员登录态。

如果令牌泄露，优先做的是轮换或吊销，而不是修改管理员密码。

## 推荐工作流

### 场景一：节点上已经跑出工作负载

1. 打开“人员”页。
2. 选择“从服务器用户开始”。
3. 选择当前已经出现的系统用户名。
4. 审核系统给出的相似账号建议。
5. 如果涉及已有绑定迁移，显式确认后再提交。

### 场景二：先给新成员建档

1. 打开“人员”页。
2. 选择手动创建。
3. 先填写显示名称和联系方式。
4. 等这个人后续在节点上出现真实系统用户名后，再补绑定。

### 场景三：未归属账号已经很多

1. 打开“人员”页。
2. 进入添加人员页。
3. 选择“一键添加未归属用户”。
4. 等待批量处理完成。
5. 在结果报告页里检查哪些用户名被自动创建、哪些被复用、哪些因为同名冲突被跳过。

### 场景四：给成员开个人移动端

1. 进入人员详情页。
2. 创建令牌并把明文令牌发给对方。
3. 如需失效旧令牌，使用轮换或吊销。

## 当前边界

- Person 数据是可选层，没有人员档案时，监控和调度仍然正常。
- 个人令牌和管理员 JWT 是两套独立认证体系。
- 服务端不会因为有了人员档案就替 Agent 改写本地调度逻辑。
- 没有绑定关系的系统用户名仍然会以原始系统用户维度出现，而不是强行猜测归属。
