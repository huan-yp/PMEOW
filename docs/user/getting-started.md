# 用户快速开始

- 启动 Web 服务端。
- 完成首次登录。
- 接入第一台节点。
- 在 Web 控制台看到实时状态。

## 先理解当前运行模型

PMEOW 当前采用两层运行模型：

- Web 服务端负责监控汇聚、告警、钩子、Agent 接入、任务镜像和最小控制面。
- Python Agent 运行在计算节点本地，负责指标采集、GPU 归属识别、本地任务队列和自主调度。

这意味着服务端是观察者和干预点，但不负责替节点做二次排队调度。任务提交和本地执行仍然以 Agent 为中心。

## 环境要求

- Node.js 20+
- pnpm 9+ 或 npm 10+
- Python 3.10+，仅在要接入 Agent 节点时需要
- Linux 计算节点，GPU 相关功能依赖 `nvidia-smi`

## 方式一：本地开发模式快速试跑

如果你只是想先看 UI 和 API 是否能正常工作，推荐用开发模式：

```bash
pnpm install

# 终端 1
pnpm dev:web

# 终端 2
pnpm dev:ui
```

开发模式下：

- Web 后端默认监听 `17200`
- UI 开发服务器默认监听 `5173`
- 浏览器访问 `http://localhost:5173`

首次打开页面时，系统会要求输入一个密码。当前实现里，如果数据库中还没有密码，这次输入会被视为初始化密码。

## 方式二：用 Docker 直接跑一个可持久化实例

如果你更关心接近生产环境的启动方式，直接运行：

```bash
docker compose up -d
```

默认行为：

- 服务监听 `http://localhost:17200`
- SQLite 数据库存放在容器内 `/data/monitor.db`
- Compose 会把宿主机 `~/.ssh` 只读挂载到容器，用于 SSH 节点采集

## 接入第一台节点

你可以用两种方式接入节点。

### 方式 A：SSH 节点

适合还没有安装 Agent 的机器。

1. 打开 Web 控制台中的“服务器”页面。
2. 新增服务器记录，填写名称、主机地址、端口、用户名和 SSH 私钥路径。
3. 点击“测试连接”，确认服务端能够通过 SSH 读取目标主机。
4. 保存后等待一次采集周期，回到“概览”页确认卡片开始刷新。

### 方式 B：Agent 节点

适合需要本地任务队列、自主调度和 GPU 归属追踪的机器。

1. 先在 Web 控制台里创建对应的服务器记录，`host` 最好直接写成节点的 hostname。
2. 在目标机器安装 Agent，并设置 `PMEOW_SERVER_URL`。
3. 启动 `pmeow-agent daemon`。
4. 等待 Agent 通过 `/agent` namespace 建立连接。

如果服务端在 `servers.host` 中找到了唯一精确匹配的 hostname，节点会自动切换到 Agent 模式，并开始显示任务队列与 GPU allocation 数据。

## 最小验证清单

完成第一次试跑后，至少确认下面几点：

- 能够打开 Web 页面并完成登录。
- “概览”页能看到服务器数量和在线状态。
- SSH 节点能够看到 CPU、内存、磁盘和网络数据。
- Agent 节点能够看到 `Tasks` 页面中的队列信息。
- Agent 节点的“服务器详情”页会出现 `Tasks` 标签页。

## 第一次上线的推荐顺序

为了减少排障复杂度，建议按这个顺序推进：

1. 先单独启动 Web 服务端并完成登录。
2. 再接入一台 SSH 节点，确认基础监控链路正常。
3. 然后接入一台 Agent 节点，确认 hostname 绑定成功。
4. 最后再启用任务队列、安全审计、钩子规则和告警抑制等进阶能力。

如果你已经准备把 Agent 部署到真实节点，继续阅读 [agent-nodes.md](agent-nodes.md)。如果你更关心服务端部署方式、数据路径和认证边界，继续阅读 [web-server.md](web-server.md)。
